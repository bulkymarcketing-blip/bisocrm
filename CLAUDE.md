## Hard conventions (FRONTEND ‚Äî these are real iOS / runtime constraints)

- **HTML via string concatenation, NEVER nested template literals.** Nested template literals crash silently on **iOS Safari** (Kushan‚Äôs test device). Hard rule for `index.html`. *(This rule is frontend-only; it does NOT apply to the backend repo‚Äôs Node code.)*
- **No native dialogs** (`alert` / `confirm` / `prompt`) ‚Äî use **`showConfirm`**; user-facing notifications via **`notif`**.
- **Two `.overlay` elements at the same z-index: an opener MUST close any other currently-open overlay first.** With equal z-index, paint order = DOM order ‚Äî a later-in-DOM overlay paints OVER an earlier one. This is why `openOverviewForm` must `CM('detail-modal')` before `OM('lead-modal')` ‚Äî `#lead-modal` is earlier in the DOM than `#detail-modal`, so the detail view covered the form (the "Edit in Overview Form" button looked dead). When adding any new overlay-opener reachable from another open overlay, check DOM order / z-index.
- **Wrap any multi-write save in `window._saving = true/false`** ‚Äî a 15-second polling loop overwrites local state otherwise.
- **Red = act-now only** (overdue, errors, danger). Never use red for informational urgency ‚Äî amber for time cues, neutral otherwise.
- Fonts: **Cinzel** (brand wordmark + quote/invoice/receipt documents ONLY) + **Inter** (everything else, incl. all screen/page/section titles, data and numbers). Navy / gold palette.

## Audit ‚Äî MANDATORY before every change (any FAIL = hard stop, investigate)

- **CSS-only changes:** prove scope ‚Äî every changed line sits inside the app `<style>` block; `{` / `}` counts unchanged; the file OUTSIDE the CSS is byte-identical (md5 the head region and the tail region before/after); the CSS class-set is unchanged.
- **JS render changes:** `node --check` on the file PLUS a **runtime sim** ‚Äî mount the changed render function with a mocked DOM + mocked data, assert it doesn‚Äôt throw and returns sane HTML, then regression-diff the untouched functions. **No sim harness is in the repo** ‚Äî build a small one for the touched screen and **commit it** so later work reuses it.
- Static check should include the overlapping-Firebase-paths rule.
- `docCSS` contains CSS braces ‚Äî **mock it** in any brace-match/extraction audit or it breaks the extractor.
- After shipping: deploy is automatic on push to `main`; **test on iPhone in a Chrome Incognito tab** (bypasses the PWA service-worker cache).

## Quote / item / installment-schedule structure (FRONTEND OWNS this contract ‚Äî Feature #5)

A quote/invoice keeps its shape: `{ id, date, expiry?, items, notes, convertedToInvoice?, sent?, advance?(inv), payments?(inv), schedule? }`. **`items` is a flat keyed map** `{ key ‚Üí item }`. Feature #5 added two OPTIONAL item fields and one OPTIONAL doc field ‚Äî **nothing existing was renamed or removed**:

- **Item** (unchanged core): `{ key, desc, qty, unit, disc(%), linkedTo?, custom?, auto?, orderDiscount?, discType?, discValue? }`. `cLine(it)=(qty||1)*(unit||0)*(1-(disc||0)/100)`; `qTot=Œ£ cLine`.
- **Package PARENT** (NEW, optional): `isPackage:true` + `pkgCeremony:'wedding'|'homecoming'|'goingAway'|'engagement'|''` (`''`/absent = "Additional Services"). It is a **$0 grouping header** (`qty:1, unit:0, disc:0`), so `cLine(parent)=0` and `qTot`/Finance/refunds are untouched. **`pkgCeremony` is deliberately NOT `linkedTo`** ‚Äî packages never collide with the event-binding helpers (`findLinked`/`syncQuoteItems` key off `linkedTo`).
- **Package CHILD** (NEW, optional): a normal item plus `parentKey:<parent key>`. Carries the full item field set; package total = Œ£ children. A package is **pinned to ONE ceremony** ‚Äî `docSectionFor` resolves a child via `parentKey` FIRST, so a child is locked to the parent's `pkgCeremony` everywhere (a child can never display under a ceremony different from its parent, even if it somehow carried its own `linkedTo`); children are positive priced lines (package/order discounts use the existing top-level discount mechanisms, not negative child lines). **Order/package discount netting:** `docSections` pulls every discount line (`linkedTo==='discount'` / `auto` / negative `cLine`) OUT of its ceremony group and sums them into a single `discountAmt`, so no ceremony subtotal is reduced by a discount; `docTotals` then renders that as one combined **"Package discount"** row beneath the Subtotal (`grand = subtotal ‚àí discountAmt`). **Document rendering:** inside a ceremony group the package renders as a light, muted **name-only** sub-label (`.pkglabel`) with its children listed beneath (`.item.pkgsub`); there is **no per-package subtotal in the document** ‚Äî the only subtotal shown is the per-ceremony one. (The Overview-Form editor does show a live package total as an editing aid.)
- **`schedule`** (NEW, optional, on the quote AND carried to the invoice): `{ mode:'custom', installments:[ { id, label, amount, due } ] }` where `due` = `{type:'confirmation'}` | `{type:'date', date:'YYYY-MM-DD'}` | `{type:'beforeWedding', days:N}`. **ONE schedule concept:** `docSchedule` renders the custom installments when `schedule.mode==='custom'`, otherwise the existing auto bespeaking+balance schedule (unchanged). Absent ‚áí auto. Invoices mark each installment Paid/Partial/Due/Overdue via a paid-waterfall. **One plan per quote** (`OF.draft.quoteSchedule`, a single object). The Overview-Form schedule editor **validates the installments against the order-discount-netted grand total** (`ofScheduleGrand` = `ofSubtotalExclDiscount() ‚àí _ofOrderDiscAmt()`): when they don't reconcile it shows a non-blocking amber "Installments total Rs X ¬∑ Rs Y under/over the quote total" warning (the schedule can still save ‚Äî it's a reconciliation cue, not a hard gate).
- **Backward compatibility:** a legacy flat item (no `isPackage`/`parentKey`) and a doc with no `schedule` render byte-for-byte as before ‚Äî proven in `sim/quoteItems.sim.js` (HEAD-vs-WORK identical HTML on legacy data). Consumers updated in the same build: `docSectionFor`, `docSections`, `docSchedule`, `docCSS`, `buildPDF`/`prevAdv`, `convInv`, the Overview-Form `buildQuoteSection` + new `qiAddPackage`/`qiAddSubLine`/`qiSetPkgCeremony`/`_qi*`/`ofSchedule*` helpers, `ofSave`/`openOverviewForm`. Untouched (regression-diffed byte-identical): `cLine`, `qTot`, `invPaid`, `findLinked`, `upsertLinked`, `removeLinked`, `syncQuoteItems`, `recomputeOrderDiscount`, `ofDiscItemKey`, `ofSubtotalExclDiscount`, `docTotals`, `buildReceipt`. Sim: `node sim/quoteItems.sim.js`.

> **CROSS-REPO (carry to the backend repo):** the backend **`getBrideQuote(token)`** reads this same `items` map and mirrors `docSections`/`docSchedule`. Before the **Bride Portal** goes live, the backend must add handling for `isPackage` / `pkgCeremony` / `parentKey` (group package children under the parent's ceremony, parent is a $0 header) and for the optional `schedule` (custom installments vs. the auto bespeaking+balance fallback), then re-verify its mirror against the structure above. **Do NOT launch the bride portal until that reconciliation is done.**

## Contracts the frontend DEPENDS ON (the backend owns these ‚Äî match them exactly)

- **`/leadintake/{id}`** (the backend / chatbot writes; the frontend reads it in `openOverviewForm`): `id` (**MUST equal the `/leadintake` key ‚Äî it‚Äôs reused as the bride id and as the delete path**), `name`, `phone` (**digits incl. country code, NO leading `+`**), `email`, `source`, `notes`, `weddingDate`, `events`, plus `nextReminderAt`, `remindersStopped`, `contactAttempts`, `stage`, `intakeAt`, `createdAt`. Review & Accept opens the Overview Form prefilled ‚Üí saving promotes to `/brides` and deletes the intake record; Dismiss deletes it.
- **`normPhone`** (must match the backend‚Äôs rule exactly; default country code 94): strip non-digits ‚Üí if len ‚â• 11 and not leading `0`, use as-is ‚Üí leading `0` ‚Üí replace the leading 0 with `94` ‚Üí 9 digits ‚Üí prepend `94`.
- **`/settings` keys** the frontend owns and the backend may read: `defaultNotes`, `logoUrl`, `defaultBespeaking`, `balanceDueDays` (default 14 ‚Äî when a balance is due/overdue), `bankDetails`, plus WhatsApp flags `waBotEnabled`, `waReGreetAfterDays`, `waAutoGreeting`, and `brideGreeting`.
- **WhatsApp inbox (#6 UI)** reads the backend-persisted thread: `/messages/{normalizedPhone}/{msgId}` ‚Äî message shape `{ id, dir:'in'|'out', kind:'text'|'image'|'template', body, mediaUrl, templateName, at (ISO), by, waMessageId, status }` ‚Äî and conversation state `/conversations/{normalizedPhone}` (`mode:'bot'|'human'`, `takenOverBy`, etc.). The reply box is enabled only inside the 24h window (else templates) and calls the backend callable **`waReply`**. Conversations are keyed by **normalized phone, NOT brideId**.
- **Bride Portal** (`bride-portal.html`, separate page, demo-mode built): consumes the backend `getBrideQuote(token)` / `acceptBrideQuote(token)` contract. **Don‚Äôt wire the CRM share button until the page is hosted** (avoids 404 links). The backend confirmed the following as-built ‚Äî the portal page / share button must honour them:
  - **Token minting:** the share button's `shareToken` must be URL-safe `[A-Za-z0-9_-]`, **16‚Äì128 chars**, or `getBrideQuote`/`acceptBrideQuote` reject it **`400 bad_token`**.
  - **Page fetch handling** (`bride-portal.html` branches on the HTTP contract): `getBrideQuote` **`200` = the sanitized payload object DIRECTLY** (no `{ok}` wrapper); an **expired** quote still returns **`200` with `expired:true`** (a gentle state ‚Äî Accept disabled, NOT a 4xx); **`404`** = invalid link; **`429`** = "try again shortly". `acceptBrideQuote`: **`200 {ok:true,‚Ä¶}`** (`justAccepted` or `alreadyAccepted`); **`409 {error:"expired"}`**; **`429`**.
  - **Notifications:** `acceptBrideQuote` writes `/notifications/{uid}` `type:'quote_accepted'` (carries `brideId`) for **every staff uid** ‚Üí shows in the CRM bell/feed. No special handling needed ‚Äî just be aware the new feed type exists.
  - **Payload shapes (post-Cycle-C ‚Äî `bride-portal.html` must render both variants before any package / custom-schedule quote is shared; flat/auto quotes are unchanged):** a section `rows[]` entry is EITHER a line `{desc, qtyNote, amount}` OR a package group `{package:'<label>', lines:[{desc, qtyNote, amount}]}`; `schedule` is EITHER `{mode:'auto', depositLabel, depositNote, deposit, balanceNote, balance}` OR `{mode:'custom', installments:[{label, note, amount}]}` (both carry a `mode` field).

### Phase 4 ‚Äî component vocabulary (introduced in `rDailyBrief`, REUSE in Phases 5‚Äì6)

Presentation-only rebuild ‚Äî every data source/action was preserved (the compute helpers `computeTodaysActions` and `_brief*` were untouched). The reusable list primitives, all in the app `<script>` just above `rDailyBrief`, are the shared vocabulary for the next screens:

- `_briefList(rows)` ‚Äî **flat list on the canvas**: emits the rows directly, separated by a 1px `--border` hairline (last row none). **No `--surf` fill, no border box, no radius, no shadow** (the concept's "flat grouped lists, not cards"). Reused by every Brief section.
