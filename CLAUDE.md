## Hard conventions (FRONTEND — these are real iOS / runtime constraints)

- **HTML via string concatenation, NEVER nested template literals.** Nested template literals crash silently on **iOS Safari** (Kushan’s test device). Hard rule for `index.html`. *(This rule is frontend-only; it does NOT apply to the backend repo’s Node code.)*
- **No native dialogs** (`alert` / `confirm` / `prompt`) — use **`showConfirm`**; user-facing notifications via **`notif`**.
- **Wrap any multi-write save in `window._saving = true/false`** — a 15-second polling loop overwrites local state otherwise.
- **Red = act-now only** (overdue, errors, danger). Never use red for informational urgency — amber for time cues, neutral otherwise.
- Fonts: **Cinzel** (brand wordmark + quote/invoice/receipt documents ONLY) + **Inter** (everything else, incl. all screen/page/section titles, data and numbers). Navy / gold palette.

## Audit — MANDATORY before every change (any FAIL = hard stop, investigate)

- **CSS-only changes:** prove scope — every changed line sits inside the app `<style>` block; `{` / `}` counts unchanged; the file OUTSIDE the CSS is byte-identical (md5 the head region and the tail region before/after); the CSS class-set is unchanged.
- **JS render changes:** `node --check` on the file PLUS a **runtime sim** — mount the changed render function with a mocked DOM + mocked data, assert it doesn’t throw and returns sane HTML, then regression-diff the untouched functions. **No sim harness is in the repo** — build a small one for the touched screen and **commit it** so later work reuses it.
- Static check should include the overlapping-Firebase-paths rule.
- `docCSS` contains CSS braces — **mock it** in any brace-match/extraction audit or it breaks the extractor.
- After shipping: deploy is automatic on push to `main`; **test on iPhone in a Chrome Incognito tab** (bypasses the PWA service-worker cache).

## Quote / item / installment-schedule structure (FRONTEND OWNS this contract — Feature #5)

A quote/invoice keeps its shape: `{ id, date, expiry?, items, notes, convertedToInvoice?, sent?, advance?(inv), payments?(inv), schedule? }`. **`items` is a flat keyed map** `{ key → item }`. Feature #5 added two OPTIONAL item fields and one OPTIONAL doc field — **nothing existing was renamed or removed**:

- **Item** (unchanged core): `{ key, desc, qty, unit, disc(%), linkedTo?, custom?, auto?, orderDiscount?, discType?, discValue? }`. `cLine(it)=(qty||1)*(unit||0)*(1-(disc||0)/100)`; `qTot=Σ cLine`.
- **Package PARENT** (NEW, optional): `isPackage:true` + `pkgCeremony:'wedding'|'homecoming'|'goingAway'|'engagement'|''` (`''`/absent = "Additional Services"). It is a **$0 grouping header** (`qty:1, unit:0, disc:0`), so `cLine(parent)=0` and `qTot`/Finance/refunds are untouched. **`pkgCeremony` is deliberately NOT `linkedTo`** — packages never collide with the event-binding helpers (`findLinked`/`syncQuoteItems` key off `linkedTo`).
- **Package CHILD** (NEW, optional): a normal item plus `parentKey:<parent key>`. Carries the full item field set; package total = Σ children. A package is **pinned to ONE ceremony** — `docSectionFor` resolves a child via `parentKey` FIRST, so a child is locked to the parent's `pkgCeremony` everywhere (a child can never display under a ceremony different from its parent, even if it somehow carried its own `linkedTo`); children are positive priced lines (package/order discounts use the existing top-level discount mechanisms, not negative child lines). **Document rendering:** inside a ceremony group the package renders as a light, muted **name-only** sub-label (`.pkglabel`) with its children listed beneath (`.item.pkgsub`); there is **no per-package subtotal in the document** — the only subtotal shown is the per-ceremony one. (The Overview-Form editor does show a live package total as an editing aid.)
- **`schedule`** (NEW, optional, on the quote AND carried to the invoice): `{ mode:'custom', installments:[ { id, label, amount, due } ] }` where `due` = `{type:'confirmation'}` | `{type:'date', date:'YYYY-MM-DD'}` | `{type:'beforeWedding', days:N}`. **ONE schedule concept:** `docSchedule` renders the custom installments when `schedule.mode==='custom'`, otherwise the existing auto bespeaking+balance schedule (unchanged). Absent ⇒ auto. Invoices mark each installment Paid/Partial/Due/Overdue via a paid-waterfall.
- **Backward compatibility:** a legacy flat item (no `isPackage`/`parentKey`) and a doc with no `schedule` render byte-for-byte as before — proven in `sim/quoteItems.sim.js` (HEAD-vs-WORK identical HTML on legacy data). Consumers updated in the same build: `docSectionFor`, `docSections`, `docSchedule`, `docCSS`, `buildPDF`/`prevAdv`, `convInv`, the Overview-Form `buildQuoteSection` + new `qiAddPackage`/`qiAddSubLine`/`qiSetPkgCeremony`/`_qi*`/`ofSchedule*` helpers, `ofSave`/`openOverviewForm`. Untouched (regression-diffed byte-identical): `cLine`, `qTot`, `invPaid`, `findLinked`, `upsertLinked`, `removeLinked`, `syncQuoteItems`, `recomputeOrderDiscount`, `ofDiscItemKey`, `ofSubtotalExclDiscount`, `docTotals`, `buildReceipt`. Sim: `node sim/quoteItems.sim.js`.

> **CROSS-REPO (carry to the backend repo):** the backend **`getBrideQuote(token)`** reads this same `items` map and mirrors `docSections`/`docSchedule`. Before the **Bride Portal** goes live, the backend must add handling for `isPackage` / `pkgCeremony` / `parentKey` (group package children under the parent's ceremony, parent is a $0 header) and for the optional `schedule` (custom installments vs. the auto bespeaking+balance fallback), then re-verify its mirror against the structure above. **Do NOT launch the bride portal until that reconciliation is done.**

## Data model (paths the frontend reads / writes)

`/brides` (pipeline + confirmed clients; a bride’s quotes / invoices / payments live underneath), `/leads` + `/customers` (phone-matched mirrors written by `convInv`), `/leadintake` (incoming enquiries — see contract), `/roles` (**read-only here; never written from the app** — role lives here, not in `/profiles`), `/profiles/{uid}` (self-editable display name), `/settings` (Firebase-synced app settings), `/counters` (doc numbers), `/notifications/{uid}`, `/notificationPrefs/{uid}`, `/fcmTokens/{uid}`, and the WhatsApp paths `/messages/{normalizedPhone}` + `/conversations/{normalizedPhone}` (for the #6 UI).

## Contracts the frontend DEPENDS ON (the backend owns these — match them exactly)

- **`/leadintake/{id}`** (the backend / chatbot writes; the frontend reads it in `openOverviewForm`): `id` (**MUST equal the `/leadintake` key — it’s reused as the bride id and as the delete path**), `name`, `phone` (**digits incl. country code, NO leading `+`**), `email`, `source`, `notes`, `weddingDate`, `events`, plus `nextReminderAt`, `remindersStopped`, `contactAttempts`, `stage`, `intakeAt`, `createdAt`. Review & Accept opens the Overview Form prefilled → saving promotes to `/brides` and deletes the intake record; Dismiss deletes it.
- **`normPhone`** (must match the backend’s rule exactly; default country code 94): strip non-digits → if len ≥ 11 and not leading `0`, use as-is → leading `0` → replace the leading 0 with `94` → 9 digits → prepend `94`.
- **`/settings` keys** the frontend owns and the backend may read: `defaultNotes`, `logoUrl`, `defaultBespeaking`, `balanceDueDays` (default 14 — when a balance is due/overdue), `bankDetails`, plus WhatsApp flags `waBotEnabled`, `waReGreetAfterDays`, `waAutoGreeting`, and `brideGreeting`.
- **WhatsApp inbox (#6 UI)** reads the backend-persisted thread: `/messages/{normalizedPhone}/{msgId}` — message shape `{ id, dir:'in'|'out', kind:'text'|'image'|'template', body, mediaUrl, templateName, at (ISO), by, waMessageId, status }` — and conversation state `/conversations/{normalizedPhone}` (`mode:'bot'|'human'`, `takenOverBy`, etc.). The reply box is enabled only inside the 24h window (else templates) and calls the backend callable **`waReply`**. Conversations are keyed by **normalized phone, NOT brideId**.
- **Bride Portal** (`bride-portal.html`, separate page, demo-mode built): consumes the backend `getBrideQuote(token)` / `acceptBrideQuote(token)` contract. **Don’t wire the CRM share button until the page is hosted** (avoids 404 links).

### Phase 4 — component vocabulary (introduced in `rDailyBrief`, REUSE in Phases 5–6)

Presentation-only rebuild — every data source/action was preserved (the compute helpers `computeTodaysActions` and `_brief*` were untouched). The reusable list primitives, all in the app `<script>` just above `rDailyBrief`, are the shared vocabulary for the next screens:

- `_briefSection(title, {count, meta}, bodyHtml)` — sentence-case Inter section header + muted tabular count + optional right-aligned meta, then the body.
- `_briefList(rows)` — **flat list on the canvas**: emits the rows directly, separated by a 1px `--border` hairline (last row none). **No `--surf` fill, no border box, no radius, no shadow** (the concept's "flat grouped lists, not cards"). Reused by every Brief section.
- `_briefRow(dotColor, mainHtml, rightHtml, onclick)` — clean tappable row (`padding:12px 2px`, sits flush under its section header): leading dot + main + trailing quick-actions. `_briefMain(name, sub)` builds the two-line label; `_briefDot(color)` the dot; `_briefEmpty(text)` is a **plain muted line** (not a boxed card).
- `_apptTypeDot(reason)` — COOL, non-gold/non-red event-type dot (consultation blue `#5B7FA6`, trial teal `#3E8E8A`, pickup slate `#7A6E8E`, else `--muted`).
- `_briefMatchesQ(name)` — Brief-scoped name filter against the global `q` (the topbar search). Real cross-record global search is a separate later capability.
- `setPT(v)` — sets the shared topbar title element (`#PT`). On `dailyBrief` it renders the **BISO brand mark** (`textContent='BISO'`, class `.brand-mark` — Inter, `letter-spacing:4px`, navy); on every other view it uses `TITLES[v]` with class `.page-title`. **Routed through `sv()` and `render()` (incl. boot)** — all `#PT` writes must go through `setPT` so the brand-mark/title swap and the boot title both stay correct.
- **Calm Brief-only action classes:** `.bicon` (borderless icon button, ~40px tap target, grey glyph) for WhatsApp/Call, and `.bact` (small navy **ghost** chip) for Log / Create quote / Open. Navy only — gold stays the topbar `+`. **Pipeline's `.lcard-icon-btn`/`.lcard-cta-btn` are a SEPARATE set and were left untouched** — don't restyle those globally; add screen-scoped classes instead.
- **Hide-empty pattern:** render a whole section only when its `count > 0` (no empty-state card), e.g. "This week's weddings/appointments". Reuse this for sparse Phase-5/6 sections.
- **Filter pills (Daily Brief only):** a row of four pills — **All / Leads / Quotes / Visits** — sits after the "N things need you today" line (CSS `.fpills`/`.fpill`/`.fpill.on`; inactive = muted ghost, **active = gold** `.fpill.on` navy-on-gold, the sanctioned active-state gold that coexists with the topbar `+`). Active filter is a **session var `window._briefFilter`** (default `'all'`; persists across in-app navigation, resets to All on page reload — no storage); `setBriefFilter(f)` sets it and re-renders `rDailyBrief`. **Section→pill mapping** (a section shows when `f==='all'` or `f===` its pill): **Leads** → Follow-ups / New enquiries / Needs attention; **Quotes** → Quotes to send; **Visits** → Appointments to confirm / This week's weddings / This week's appointments; **All-only (no pill)** → Payments / Approval requests / Since last visit. Empty sections stay hidden; the topbar search still filters rows within a shown section; the "N things" count stays the unfiltered total. Empty filter ⇒ a quiet `_briefEmpty('Nothing under <Pill> today')` line.
- **`renderTodaysActions(actions, filter)`** now takes an **optional `filter` (defaults `'all'`)** that gates its four sub-groups (Follow-ups→`leads`, Appointments to confirm→`visits`, Quotes to send→`quotes`, Payments→`all`-only). The **Daily Brief passes the active pill**; **Pipeline (`rPipeline`) passes nothing → `'all'` → output byte-identical to before** (the shared widget is unaffected — proven in the sim). Don't make Pipeline pass a filter.
- Brief row CTAs stay **navy** (the calm `.bact`/`.bicon` above) so the only gold on the Brief is the topbar `+`. **Flag for Phase 5:** Pipeline cards still use gold WhatsApp CTAs (`btn-gold`), so with the gold `+` there are competing golds on Pipeline until Phase 5 harmonizes them.
